Error handling
 
All API functions that have a return value return MT_OK_ERR upon success. If the return value is MT_iO_ERR, it indicates that there is a problem with the network connection or serial port connection. You should first call CloseReader and check these connections before attempting to call InitReader_Notype. As for MT_CMD_FAILED_ERR, it only means that the function has failed, it is not a fatal error, and you can do anything next. But if it appears in the inventory, it is usually due to abnormal communication data. Frequent occurrences indicate poor communication conditions. As for MT_CMD_NO_TAG_ERR, it generally cannot be referred to as an error, indicating that no tag was found during the tag operation.
  
Errors starting with MT_HARDWARE_ALERT_ERR_BY are serious errors that cannot be ignored and may cause hardware damage. There are some incorrect operations that can cause these errors, including:
  • tag operation on reader antenna port without antenna connection
  • Using unqualified antennas
  • The ambient temperature is too high
  • The antenna is located very close to the metal in front, which can cause excessive return loss
It is best to first turn off the reader and check the working conditions and environment when these errors occur.
 
There is another way to troubleshoot errors, where you can immediately call the GetLastDetailError function after an error occurs to obtain more detailed error codes and error messages represented by strings.

When calling the GetLastDetailError interface, the following rules should be observed:

  • After InitReader_Notype returns a failure, the GetLastDetailErrorinterface should ​​not​​ be called, as the handle hReaderis invalid at this point.
  • If an RFID operation (such as read, write, lock, or parameter configuration) returns a failure (i.e., any value other than MT_OK_ERR), the GetLastDetailError interface ​​may​​ be called.
  • The GetLastDetailError interface ​​can​​ be called within the callback function ReadingErrorHandler.
    •  
    The current version of the SDK does not guarantee thread safety for all functions (except for the InitReader_Notype function) when operating on the same reader instance handle. Users must ensure that there are no race conditions in API function calls or guarantee mutual exclusion when calling API functions across multiple threads. These restrictions do not apply to different reader instance handles.
